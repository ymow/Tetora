単一エージェントに質問に答えさせるのは便利だ。でも、リサーチ・ドラフト・レビュー・公開を自動的にこなすエージェントのシーケンスを動かせるなら、それは変革をもたらす力になる。Tetora のワークフローシステムがそれを可能にする。

## ワンショットプロンプトの問題

タスクに複数のフェーズがある場合、一つのエージェントがすべてを処理しようとすると、予測可能な形で問題が起きる：コンテキストのオーバーフロー、早期の判断が後の選択を制限する、そしてどこかのステップが失敗しても綺麗にリカバリできない。

より良いアプローチ：作業をフェーズに分割し、各フェーズに集中したエージェントを割り当て、各ステップの出力を次のステップの入力にする。

## マルチステップワークフローの定義

Tetora のワークフローは、各ステップに明確な役割・入力・出力が定義されたタスクのシーケンスだ。以下はシンプルな例：

```json
{
  "workflow": "weekly-report",
  "steps": [
    {
      "id": "research",
      "agent": "hisui",
      "prompt": "過去7日間のAIニュースのトップ5を収集して。title、url、一行サマリーを含むJSON配列で出力してください。",
      "output_key": "news_items"
    },
    {
      "id": "draft",
      "agent": "kohaku",
      "prompt": "{{news_items}} のニュースを使って300字のダイジェストをMarkdown形式で書いてください。",
      "depends_on": ["research"],
      "output_key": "draft_md"
    },
    {
      "id": "review",
      "agent": "ruri",
      "prompt": "{{draft_md}} のドラフトを正確さとトーンの観点でレビューして、最終版を返してください。",
      "depends_on": ["draft"]
    }
  ]
}
```

`depends_on` フィールドは、上流のステップが正常に完了するまで、そのステップを開始しないよう Tetora に指示する。`research` が失敗すれば `draft` は実行されない——トークンを無駄にせず、壊れた出力も生まない。

## ステップ間でコンテキストを渡す

各ステップの `output_key` の値は、下流のステップで使えるテンプレート変数になる。後続のプロンプトで `{{key_name}}` を使えば、前のステップの出力をインラインで注入できる。

これにより、各エージェントのプロンプトは短く集中したものに保たれる——エージェントは必要なものだけを読み込み、それ以前に起きたすべてのことを把握する必要はない。

## ワークフローの実行

```bash
tetora run workflow weekly-report
```

Tetora は依存関係の順序でステップを実行する。共通の依存がない独立したステップは自動的に並列実行される。各ステップの最終出力はレビュー用にタスクレコードへ記録される。

## ヒント：失敗からのリカバリを想定して設計する

ステップに分かりやすい名前をつけ、各ステップのスコープを狭く保とう。`draft` が失敗した場合、`research` をやり直さずに単独で再実行できる：

```bash
tetora run workflow weekly-report --from draft
```

ステップが短いほど失敗が早くわかり、リカバリのコストも低い。

## まとめ

マルチステップワークフローは、一人のオペレーターがスケールする方法だ。フェーズを定義し、依存関係を繋ぎ、オーケストレーションは Tetora に任せよう。一度安定して動いたワークフローは、毎回安定して動く。
