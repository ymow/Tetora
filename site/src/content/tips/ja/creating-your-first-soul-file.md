## 課題：毎回の応答がバラバラに聞こえる

決まったアイデンティティがなければ、エージェントはブレる。あるセッションではフォーマル、次のセッションではカジュアル。トーンのガイドラインは忘れられ、複数エージェントのスタイルが混ざり合い、まるで5人の別人が書いたようなアウトプットになる。

**Soul ファイル**がこれを解決する。エージェントの個性・声・行動の制約を固定する Markdown ドキュメントで、セッション開始時にロードされ、全体を通じて参照される。

## Soul ファイルに書くこと

Soul ファイルは4つの領域をカバーする：

1. **アイデンティティ** — 名前、役割、エージェントが大切にすること
2. **ボイス** — 表現スタイル、言語習慣、避けるべきこと
3. **関係性** — チームメンバーとの連携方法
4. **制約** — ハードルール（絶対にXしない、常にYする）

## セットアップ手順

Tetora ワークスペース内に `agents/{name}/SOUL.md` を作成する：

```bash
mkdir -p agents/kohaku
touch agents/kohaku/SOUL.md
```

最小限の Soul ファイルの例：

```markdown
# SOUL.md — 琥珀

## アイデンティティ
- 名前：琥珀
- 役割：コンテンツ担当——投稿、記事、SNSコピーを書く
- 核心的な信念：良いアイデアは聞かれる価値がある。私の仕事はそれを届けること。

## ボイス
- 温かく、率直で、ストーリー優先
- 比喩を使って技術的な概念をわかりやすくする
- 専門用語は使わない。使う場合は必ず説明する
- 主言語：繁体字中国語。英語・日本語を自然に混ぜる

## 関係性
- 琉璃（リーダー）：最終承認権限——琉璃がNOと言ったら、それはNO
- 翡翠（リサーチ）：主なインテル源——書く前に必ずレポートを読む
- 黒曜（エンジニアリング）：技術的な橋渡しが必要なとき「平易な言葉でどう表現する？」と聞く

## 制約
- 琉璃のレビューなしに公開しない
- 数字を誇張しない。釣り見出しを使わない
- すべての投稿はソースまで遡れる状態にする
```

## エージェント設定に組み込む

エージェントの設定ファイルで Soul ファイルを参照し、自動でロードされるようにする：

```json
{
  "agent": "kohaku",
  "soul": "agents/kohaku/SOUL.md",
  "model": "claude-opus-4-6",
  "role": "content"
}
```

エージェントがセッションを開始すると、Soul ファイルはシステムコンテキストに注入される。すべての応答はそこで定義されたアイデンティティを通過してフィルタリングされる。

## 効果

Soul ファイルが整ったら、トーンを修正し続ける必要はなくなる。コンテンツだけに集中できる。個性がセッションをまたいで、タスクをまたいで、コラボレーターをまたいで安定する。エージェントは汎用 AI ではなく、*自分自身*として語るようになる。

まずアイデンティティと制約から書き始め、ブレに気づいたらボイスのルールを追加する。Soul ファイルはエージェントと共に育っていく。
