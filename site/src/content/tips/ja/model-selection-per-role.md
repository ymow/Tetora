## 問題：全タスクに同じモデルを使う

すべてのエージェントを Opus で動かすのは安心感があります。でも高コストで、初稿作成やログ解析のような作業には完全にオーバースペック——推論レベルの料金でフォーマット作業をしている状態です。

逆に、戦略的な判断を Sonnet に任せると、深みに欠けるアウトプットになりがちです。

解決策は**ロール別にモデルを割り当てること**。デフォルト任せはやめましょう。

---

## Tetora のモデルロールの仕組み

Tetora の各エージェントには Soul ファイルに `role` フィールドがあります。`tetora.config.json` でロールとモデルをマッピングできます：

```json
{
  "models": {
    "draft": "claude-sonnet-4-6",
    "review": "claude-opus-4-6",
    "strategy": "claude-opus-4-6",
    "parse": "claude-haiku-4-5-20251001"
  },
  "agents": {
    "kohaku": { "role": "draft" },
    "ryuri":  { "role": "review" },
    "hisui":  { "role": "strategy" },
    "jade":   { "role": "parse" }
  }
}
```

タスクがエージェントにディスパッチされると、Tetora はそのロールに対応したモデルを自動的に使用します——毎回の設定は不要です。

---

## 実用的なマッピング表

| エージェントロール | 推奨モデル | 理由 |
|---|---|---|
| 初稿作成 | Sonnet | 高速・低コスト、生成系タスクに最適 |
| コードレビュー | Opus | diff の深い推論が必要 |
| 戦略 / 計画 | Opus | 多段階推論、盲点が少ない |
| ログ解析 / タグ付け | Haiku | 構造化抽出、速度優先 |
| 最終編集レビュー | Opus | Sonnet が見落とした点をキャッチ |

---

## タスク単位のオーバーライド

デフォルトを上書きしたい場合は、ディスパッチ時に `--model` フラグを使います：

```bash
tetora dispatch --role kohaku --model claude-opus-4-6 "ランディングページのコピーをゼロから書き直す"
```

グローバル設定を変えずに、一度きりの重要タスクに使えます。

---

## 結果

典型的なコンテンツ作業日——下書き 5 本、レビュー 2 回、ログ解析 20 件——このルーティングパターンにより、全部 Opus で動かす場合と比べてトークン消費が約 40〜60% 削減。最終アウトプットの品質に目立った差はありません。

**覚え方：Opus は意思決定、Sonnet は実行、Haiku は抽出。**
