## 問題：並列実行すべきでないステップ

ワークフローには自然な順序があります。デプロイの前にテストを走らせたい、ビルド後にセキュリティスキャンをしたい、分析が完了した場合のみレポートを投稿したい——。依存関係なしに全ステップを一度にディスパッチすると、競合が起き、デプロイ破損・偽陽性アラート・不完全なデータ上のレポートが生まれます。

Tetora では **タスクチェーンと依存関係** を使って、こうした前後関係を明示的に表現できます。

## タスクチェーンの仕組み

タスクをディスパッチする際、`dependsOn` フィールドに1つ以上のタスクIDを指定できます。Tetora のスケジューラーは依存タスクを `waiting` 状態で保持し、上流タスクが `done` になった時点でキューに追加します。

```json
// dispatch payload の例
{
  "task": "deploy-to-staging",
  "agent": "kokuyou",
  "dependsOn": ["task-abc123"],
  "onFailure": "abort"
}
```

- `dependsOn` — 先に完了すべきタスクIDの配列
- `onFailure: "abort"` — 上流タスクが失敗した場合、このステップを完全にスキップ
- `onFailure: "continue"` — 結果に関わらず実行（クリーンアップや通知ステップに有用）

## 実例：テスト → デプロイ → 通知

各ステージが前のステージを待つ3ステップパイプラインの例：

```bash
# ステップ1：テスト実行（依存なし）
tetora dispatch --task "run-test-suite" --agent hisui --output task-id

# ステップ2：テスト通過後のみデプロイ
tetora dispatch --task "deploy-staging" --agent kokuyou \
  --depends-on $(cat task-id) \
  --on-failure abort

# ステップ3：デプロイ結果に関わらずDiscord通知
tetora dispatch --task "notify-team" --agent kohaku \
  --depends-on $(cat task-id) \
  --on-failure continue
```

テストエージェントが最初に実行されます。テストが失敗すれば、デプロイは自動中止。通知ステップは常に発火します——チームはどちらの結果でも更新を受け取れます。

## なぜ重要か

GitHub Actions や Buildkite などの CI システムは、長年 workflow YAML でこれを実現してきました。Tetora は同じロジックを **エージェントネイティブのパイプライン** に持ち込みます——「ステップ」がシェルスクリプトではなく、独自のメモリ・ツール・推論能力を持つフルエージェントであるパイプラインに。

その結果：失敗に対して耐性があり、自己記録的で、各ステップのステータスが個別に追跡されるため、デバッグが容易なパイプラインを構築できます。

## クイックヒント

- ステップ数は自由——依存関係は線形リストではなくDAGを形成します
- `tetora task status <id>` で待機中・実行中・ブロック中のステップを確認
- cron トリガーと組み合わせて完全自動化のナイトリーパイプラインを構築
- `onFailure` は常に明示的に設定——デフォルトは `abort` ですが、明示した方が安全

小さく始めましょう：既存のディスパッチに1つの依存関係を追加して、パイプラインがどう自己組織化するか確認してみてください。
